INPUTS:
    cpu.a;
    cpu.pb;
    cpu.pc;
    cpu.nmi_pending;
    ppu.scanline;
    sys.frame;

VARIABLES:
    Bool f1;
    Num cnt;
    Num cnt2;
    Num prev_scanline;

INIT:
    f1 = false;
    cnt = 0;
    cnt2 = 0;

LOGIC:
    if (cpu.a == 0xFF && cpu.mem[cpu.full_pc]):
        BREAK;
    
    if (cpu.pb != 0 && cpu.pc >= 0x8000):
        f1 = true;
    
    if (f1 && cpu.nmi_pending):
        cnt = cnt + 1;
    
    if (cnt == 1000):
        cnt = 0;
        LOG("1000th NMI outside of PB: 0 && pc < 0x8000");
        BREAK;
    
    if ( (ppu.scanline & 0xFF) != prev_scanline ):
        prev_scanline = ppu.scanline & 0xFF;
        cnt2 = cnt2 + 1;
    
    if (cnt2 == 1000):
        cnt2 = 0;
        if (sys.frame > 1000):
            LOG("Scanline: ", ppu.scanline, ", Frame: ", sys.frame);
            BREAK;
    
------------------------

Types:
    Bool, Byte (u8), Word (u16), and Num (usize)
    String literals in LOG statements only.
    Types cannot be cast for now.
    Single line comments can be added using a double forward slash (//).

Keywords:
    INPUTS - Lists the inputs that the emulator will need to update for the
             logic. If an input is not listed in the INPUTS section, then it
             will result in a compile error.
    VARIABLES - Lists the variables that will be used in the logic block
                section.
    INIT - Where variables are initialized. Conditions may be checked here as
           well. In the future, there may be additional keywords for enabling
           and disabling watchpoints. This is where the ENABLE_WATCHPOINTS
           keyword would be. But if the watchpoints are only to be run before
           frame 1000, for example, then there could be a statement like this:
        
            if (sys.frame < 1000):
                ENABLE_WATCHPOINTS;
        
           This would be valid in the INIT block.
        
    LOGIC - A set of conditional statements that will be checked when their
            inputs are changed.
    BREAK - Causes the emulator to halt. Under the hood, this will store a bool
            called "fired." If the BREAK is reached, then fired becomes true.
            The fired flag will remain true until the condition that caused it
            to fire is re-evaluated and is false.
    LOG - Works similarly to BREAK, but accepts a list of string literals,
          input system variables, and script variables. The inputs will be
          concatenated at runtime and the resulting message will be logged.
          Emulation will not stop when a LOG statement is hit.
    if - Each if statement will be evaluated in order (if its inputs are
         changed).
    else - Can be paired with if statements to perform logic conditionally
           without having to type out the negation of the if condition.

Each condition statement must end with a :
Each line must otherwise end with a ;
